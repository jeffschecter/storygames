\n I've been working on FiascoPlaysets.com for a few weeks now. It is an easily browsable, mobile-friendly list of Fiasco Playsets intended to make finding, downloading, and ultimately playing any Fiasco playset as easy as possible.

So far, it only includes core playsets, playsets of the month, Bully Pulpit special releases, and a handful of promotional playsets from third parties. Over the next few weeks and months, I intend to make the list as exhaustive as humanly possible.

Please take a look, let me know what you think, and if you want to help out, submit missing playlists through the contact form at the bottom of the site. I hope this proves useful to some of you out there!\ufeff

    \n I'm surprised you need to do data entry, when the site already has everything in code. Just push it through a script to make it whatever format you need it to be (XML?) and you'd be able to put all the wiki sets on in moments.

    Do you plan to add dice rolling and, thereby, only show available options as the dice are used (with, of course, all options available for the final element needing specifics)? I've had several folks point out that such an app would be super-useful, and it would justify data duplication (and the commensurate need for sync, if playsets get changed or edited or otherwise corrected).

    Even better might be just to read from the wiki automatically, and self-assemble the playsets into the app. THAT could be a tad challenging, between walking the "Playset" Category page's links, then parsing the elements page for each (basically, add "_Elements" to the playset home page link from the Category page, then parse through # and ## lines on that page).
    A tad tricky... but a sight better than having to hand copy&paste it all from PDFs (I know: I posted several of the first playsets to the BPG wiki; and I almost considered do it with Visual Basic, just to get the text. Unfortunately, reading from PDF is a LOT harder than pulling from an HTML page, which is all you'd have to do).

    \n Basically, Alan, I'm proposing that the app be a 'skin' of the BPG wiki, not a parallel site with the exact same data in a different, hard-coded format. The dice rolling and element filtering to show only those elements that the dice permit you to choose would be an actual functional difference from the wiki, not a mere aesthetic one.

    When doing the templates for the wiki, we figured one list for all elements is easier on the database and on the average user, especially if one is trying to figure out what you need to choose early if you want it before its requisite die values became unavailable. Like, if I only have two 2s, I need to quickly scan all the 2-value element categories to see if I want one of them, and then probably scan all 2-value specific elements, to see if I want one of them badly enough to try to force it/cajole the other players into working towards it with their choices. Happens just about ever time I demo the game: there's at least one player who notices, "Hey, only a couple of 2s! What will become unavailable when they're gone, and do we want it instead?" [That player is usually me, as I try to expose some of the metagame in the rules in actual play.]

    Oh, and with that in mind, the app would probably have to show all dice results, rather than just filter 'behind the scenes'. Easy enough, though: one line of values, sorted and in incremental order, showing in the header or the footer of the app's reading area. Selecting (and verifying?) a category or element turns the number from black to red, say.
