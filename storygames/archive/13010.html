    \n Maybe I'm off base, but don't most EPUB readers handle CSS for themselves? I use Aldiko on Android, which has a TON of controls for font, colors, size, and orientation in the app, which (I'm assume) override any CSS packaged with the EPUB.

    What do you get on your various reader platforms when you just output from InDesign?

    Perhaps more importantly, how much of your content *must* be typeset in a particular style; how critical is presentation for an EPUB format of your game(s)?
    -----
    On a related note, you might find much of your whole book organization should change for EPUB, as the only common navigation controls are search, chapter selection, and scrolling.

    Search often is slow as hell (in RPG electronic docs) due to too many hits (an "overzealous indexing" problem). Scrolling is right-out, for obvious reasons (all the scanability triggers--layout, image positions, text block positions--are gone; the number of screens compared to number of pages is HUGE).

    So the best bet is chapter selection, because you can name the chapters whatever you want in the toc.ncx. As a result, EACH concept could have its own chapter, as would each discrete procedure or type of reference. At a bare minimum, chunk your source into chapters for every section and sub-section, and name and enumerate them meaningfully (but concisely) in the toc.

    HTH;
    David

    \n You will probably have to make the toc.ncx file yourself. Which could also mean that you have to re-chunk (into individual HTML files) each section, to be able to have a TOC entry call a specific section (i.e. not a whole chapter). But editing the toc.ncx file is pretty simple--it's just markup language with a couple of tags (one of which has a couple of attributes) per TOC entry. If you built the original out of text embeds or individual files per section (not per chapter, which it tries to default to) then it should be easy enough to save each off as HTML. If you didn't chunk that low-level, maybe you want to rebuild your master book file (and source section files) to use more discrete chunks, and then see ow exporting/TOCing works.

    All that said... you could probably copy & paste into new HTML files (with minimal tagging for headings) in less time than it takes to rebuild in ID.

    This all goes to show: planning for ALL potential deliverables is key, prior to building a book file and chunking content. (Even more so if you are trying to leverage any sort of single-sourcing for several pubs in a library, but that's rare in small-press RPG circles.)

    \n I'm preparing a blog post on this subject, based on stuff I tried in my aborted Game Chef attempt, Exodus. One problem I'm having is that I don't have access to many readers. So far, I've looked at the epub I made on the iPad, in the Mac version of Cailbre and in Stanza (Mac and iPad), with mixed results.

    I'd be obliged if anyone with an interest in this topic would check out the Exodus epub (see link above) in whatever readers they have access to, and tell me problems they see either here or at the link above. Screenshots would be even better.

    So far, I can see that there seems to be quite a bit of difference in CSS support among readers. Stanza, for example, centers all the section headers, even though the CSS says not to (or, perhaps, it's just that the default is different. I have to check that). Graphics support also seems to be a bit different (Stanza, for example, doesn't seem to know what to make of the graphic cover, while the others show it just fine).

    I haven't seen one that honors CSS2 properties like keep-with-next, which is sort of irritating.

    I also notice that if you italicize text using a character style, it tends to work more consistently in the resulting epub than if you just use a keyboard shortcut on the fly.

    Some notes as I see stuff:\n
      \n
    • In the first couple of pages you'd need more vertical spacing between paragraphs. They feel cramped.
      • \n
    • Careful with colors: the examples appear light grey and are just barely readable.
      • \n

      • \n
    • How many circles should I see on each line in the "Fleet Gessen" illustration? I see 9 in portrait mode and 12 in landscape.
      • \n
    \nSome suggestions:\n
      \n
    • Font embedding: to style a bit more the text consider embedding a font (for the titles, I'd say). Embedding fonts is quite easy and the readers *should* support it, generally. If they don't, it's definitely a bug.
      • \n
    • Vector illustrations: instead of using raster images, especially for the infographics, consider using SVG images. They can be embedded directly in the html or linked as external images, just like png or jpg. The big difference here is that with CSS you can set a percentile size that will always scale with the reader's screen at any zoom level, without ever losing quality due to the resizing. This would fix your problem with the size of the chart on the last page.
      • \n
    • Other embellishments like chapter separators, graphic sidebars, and so on: these, too, can be done with SVG. Just be careful not to overdo it.
      • \n
    \nFor a very basic example of SVG embedding, see this post on my blog.

    I have a for-personal-use-only homemade version of Thy Vernal Chieftains, for example, using SVG separators, embedded fonts and SVG character sheets that works much better than the pdf on my reader.

    Almost forgetting: the BeBook reader uses the Adobe "ADE" ePub reader software (with support for drm-ed epubs)... which has its known bugs and quirks, but works fine.

    \n CS5 does allow custom CSS, and my epub uses it, but only by overriding the entire sheet for the export. If, instead, you let CS5 create the CSS for you, in converts all paragraph styles into their CSS equivalents. I actually want something in-between, where you can override specific styles to tweak the CSS it generates automatically.

    As an example, the chapter titles use the Header 1 style. In InDesign, this is defined to be the large white text you see in the PDF, where white makes sense because the text hovers over the image. When exporting to epub, the format can't handle the text floating over the image, so by default it throws the text after the image. If that text remains white, it becomes unreadable, because now it is on a white background. So, I have to override the CSS to make that style use black text instead. It would be nice if CS5 allowed a way, in the definition of the "Header 1" style, that I could say, "this is white text, but when I'm exporting to EPUB, override certain things". If I was building such a feature, I'd add a section to the style editor that gave you a preview of the CSS that would be generated at export, and allow the user to edit it.

    \n What I'd suggest is... stop trying to use InDesign to do ePub. Yes, it can do the grunt work of building the package and 90% of the book, but the fine tuning stuff I'd do by hand. It's very much like trying to use dreamweaver vs doing the coding yourself.
    It's really really easy if you know HTML/CSS.

    With CSS tricks you can easily do neat stuff like Drop Caps, for example.

    A truly ideal workflow, actually, would keep the content and layout separate using some intermediate markup format (like DocBookXML) that can be imported by InDesign for the fully graphical pdf and auto-converted in ePub.

    @Lester: yes, sorry, I checked back and the grey text is the "notes to self" text, actually.

    Re: font embedding.
    Yes, it can be tricky and is currently very badly (if at all) supported by InDesign. Doing it by hand is quite simple, though. If all you need is some styling on the titles, btw, it's much safer to convert the text to path an save it as standard SVG.

    Oh, also, a general heads-up on CSS: never, ever use absolute units and measures. Percentages and ems all the way: you have no idea at all of the nature of the device that will display your text.
