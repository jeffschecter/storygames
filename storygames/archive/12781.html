    \n I'm going to echo the DokuWiki recommendation. It's what I've been using for some time.

    It's easy to set up, has a good selection of templates available - many of which are fairly easy to customize, and a very large set of plugins available to extend it's capabilities (including a plugin for WYSIWYG editing).

    I particularly like the ease with which you can lock down all, or parts, of the wiki to prevent unwanted editing.

    Also, it's a text files based wiki; so there's no messing about with an SQL database needed.

    \n My "traditional wisdom" approach to wikis have always been - if you're looking for a large number of contributors, you need to lower the barrier of entry, which means you should go with the industry standard which people know (even if they don't "love"). That pretty much means Mediawiki. I don't mind it, and while it's a bit rough around the edges it has a shit-ton of cool plugins.

    If you're going to have a small, controlled set of contributors, why not a Google Doc?

    I will follow this thread with great interest though - always interested in learning what people use and love.

    Posted By: OrangeDiceMediawiki seems like it'd be overkill if it's just a short ruleset, and IMO is the least simple.
    \nYup! Which is why I mentioned Google Docs. Two security modes (view or contribute), super-simple editing, automatic versioning (though not as robust as Mediawiki's), real-time collaborative editing, and so on. It is superior to any wiki product I've seen, assuming your set of contributors is small and your project doesn't require a whole lot of different pages with intra-wiki hyperlinks (most SRDs, whole games in fact, work perfectly fine as a single document written and read from top to bottom).

    While I appreciate all those other wiki products out there, that are more focused on usability and maintainability than Mediawiki, I find it hard to make their case; you either need something simple like Google Docs, or something much more scalable, customizable, and while complex, familiar to the user like Mediawiki. I'd love to be proven wrong, since I appreciate software heterogeneity.

    \n I'm a huge fan of semantic wikis (where the links know why they are linking, in ways that can be leveraged), but they take commitment to really get the most out of. I use the semantic plugins for MediaWiki to do this.

    As an example of why you might want to do something like this in an SRD, consider a crunchy game like Pathfinder. It would be possible to, for example, put semantic tags in descriptions of feats. Then on the "list of feats" page, instead of manually listing them all, use a semantic query to build the list automatically as feats get entered.

    I use a wiki like this to track longer campaigns (particularly those with lots of interlocking NPCs, locations and agendas). As an example, this list of characters from an Exalted campaign is completely computer generated.

    The trick to semantic wikis is that you need to start doing the semantic bits early. I added them fairly late into my campaign wiki, so semantic tagging is a bit spotty.
