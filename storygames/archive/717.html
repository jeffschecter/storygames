    El Fuego, as a strong proponent of MUSH over IRC, I'm going to play a little bit of devil's advocate and tell you that everything you describe can be implemented as an IRC bot. In fact, you could even go as far as having a bot that listens to @desc commands and keeps track of everything. Or, you could do it some other way, replacing the obscure @ thingie with something a little easier to remember for those not conversant with MUSH conventions.

    That said:

    Advantages of coded MUSH objects:
    - They run on the server and keep the state info on the server
    - They can be maintained by anyone who is online
    Disadvantages of coded MUSH objects:
    - They have to be coded in MUSHCode, which is tricky to learn, these days
    - MUSHCode reads like random white noise

    Advantages of IRC bots:
    - They can be coded in any modern programming language
    - Some IRC clients already come with good bot support
    Disadvantages of IRC bots:
    - They have to run somewhere, other than the server
    - They can only be maintained by someone who has access to the machine they run on

    Now, to me, I'm already completely fluent in MUSHCode, which means the advantages of IRC bots are nearly nil. On the other hand, the differences between IRCing and MUSHing aren't restricted to support objects:

    In support of IRCing:
    - An IRC channel is conceptually simpler than a MUSH room
    - Nowadays, resources for IRC users are more abbundant than for MUSH players
    - IRC servers are already out there, whereas a MUSH has to be created and maintained by someone

    In support of MUSHing:
    - Simultaneous games in a MUSH server are easier to set up, manage and follow than in an IRC channel
    - MUSHes can persist the game state across sessions
    - MUSH servers can be tweaked, recompiled and restarted on the fly, if the games need it

    I think all in all, a game is best served by a MUSH over IRC. IRC has some distinct advantages that we best to lose track of, however.

    I'll start. Edit: Well, I wanted to start, but Joao and Joshua beat me to it.

    IRC

    Inter Relay Chat, or IRC, is a server-based technology that lets people talk on virtual "channels" and also allows private person-to-person chat. It's been around for around 20 years (21 if you count its predecessor, Bitnet).

    You can connect to IRC via plain ol' telnet or you can download a specialized IRC client like mIRC or even Trillian. The experience is always better with a real client; telnet sucks.

    IRC provides very little server-side coding functionality. It manages channels, users and nicknames, and some privileges for channels and users. It's a virtual CB radio with security controls. People extend the functionality of IRC by writing scripted "bots" that connect like a user, listen for certain command strings, run their subroutines, then respond with text. A dice-roller bot might listen for "roll 2d6" and then respond with the sum of two random d6 rolls. mIRC has a built-in scripting language and there are some stand-alone bot programs you can run. My limited understanding of modern IRC is that all this cool code is client-side, meaning the dude with the dice scripts has to have his bot connected for you to use it.

    We gamer folks have a channel called #indierpgs on irc.magicstar.net.

    MUSH

    MUSH is said to stand for "multi-user shared hallucination," but it doesn't really. It was just a play on MUD and they made up the acronym later. MUSH is another server-based technology that creates a text-based virtual world for people to explore. A user creates one or more accounts ("players") who can log in at the same time. The interface feels like ZORK or some other Infocom or Scott Adams game on steroids. TinyMUSH, a popular type of MUSH server, has been around since 1990 or so.

    You can connect to a MUSH using plain ol' telnet or you can download a specialized MUSH client like SimpleMU. I've listed clients for other platforms elsewhere. The experience is always better with a real client; telnet sucks.

    Connected players can create rooms and give them descriptions, link them together with doors ("exits"), and create other random objects ("things"). Each object has all kinds of predefined properties that a player can set to get different kinds of behavior. It's pretty trivial to set up a network of rooms with cute (and sometimes annoying) little messages that fire off when you walk through doors.

    The game persists objects until they're deleted by a player. That means that all the game's code is available all the time. You can also persist data. This means that if you stop your Dogs game in the middle of a conflict, you can resume it a week later with your dice pools intact -- assuming the dice objects are coded that way (and that's how we're trying to code the game-specific stuff for Foundry).

    MUSH has a built-in scripting language that is very powerful, but at times frustrating. My other MUSH project is FiranMUX, which has like 600,000 lines of custom softcode for a variety of systems from clothing to combat to economics to magic. While the script can be somewhat arcane, a programmer experienced with other languages can pick it up at a reasonable level in a short amount of time.

    Experiential Differences

    For me, one big difference is the feeling of being there. On IRC and other pure-chat media like AIM, the interface feels like a virtual telephone. I'm here. My friends are there. I can hear them and talk with them. On MUSH, the interface feels like a virtual world. I'm in there. My friends are in there in the same place as me. I can see them and hear them. I can gesture ("pose") in a natural way that supports immersion (oh noes! the I-word!).

    I've been thinking bout the pro/con thing for a bit now, and I guess I should offer up something substantive...

    Clients
    IRC is an extremely popular protocol, and thus there are many, many clients for it. This is good and bad. Good because it means it is easier to find a client that is just right for you. One that has the features and presentation and aesthetics that you want. Bad because it means no one knows how to help you if you have problems outside of the most common clients.

    MUSH, being less popular, has fewer clients. This inverts the above pro/con thing. You're pretty much stuck with one of a few clients which means fewer features and options, and if you don't like what you've got: tough. However, it also means that it's extremely likely that someone already online is an expert in the use of the client in question.

    Persistent Code
    MUSH beats IRC hands-down on this one. While it's possible to code an IRC bot to do anything a MUSH object can do, the IRC bot is client side. That means that if something goes wrong on the computer of the person running that client then the code is no longer available. Since MUSH objects are server-side, once they are written and dropped they're available to anyone who wants one.

    Of course MUSH is a ridiculously archaic programming language, whereas you can code an IRC bot in whatever language you happen to know. This is actually not that big an issue for us specifically because apparently we already have a number of MUSH coders already, so no one's really forced to learn a new language.

    Interface
    IRC was originally developed as a communications tool. This means that the interface and hard-code are designed to facilitate communications first and other stuff is tacked on at the end. This means it's easy to chat (just enter a line of text and hit enter), and that there's a unified command-access character (the "/") for doing anything that's not direct chatting.

    MUSH was, as far as I can tell, originally developed as an immersive tool. This means that the interface and hard-code are designed to facilitate immersion first. This means that the command-access character goes away. I mean, it's more immersive to say "get lamp" than "/get lamp". Further, MUSH doesn't assume a default action for any line of text.

    In IRC, if text isn't somehow special, it gets said. In MUSH, if text isn't denoted as special with a command, you get an error message.

    Everything else
    Beyond that, as far as I can tell, anything you can do in a MUSH you can do in IRC and vice versa. MUSH's ":" is equivilent to IRC's "/me", and so on.

    It becomes a matter of interface preference, which in many cases boils down to experience. I've been using IRC for a long time, so certain commands are intuitive, whereas I have to think about them in MUSH. And I'm sure the same is true of old MUSH hands in IRC.

    For instance, I find it really annoying that just to say something in MUSH I need either a four key-stroke prefix ("say ") or a shift-keyed prefix ("). That slows things down unneccessarily! (So I proclaim.) I'm sure MUSH people don't even notice it anymore.

    Anyway, I'm all for growing the community, but I think it's unfortunate (not that I don't appreciate it Adam!) that we have a MUSH instead of a move to join the existing community in #indierpgs. At the moment I have to hang out in one place to talk to some people, and in the other to talk to others. Why can't we all just get along?

    Thomas
