\n (Apologies if this crosses over into Directed Promotion.)

Since last December, I realized I'm just absolute crap at designing "big" games in a purely text-based format. I get too wrapped up in the details to see how all the individual gears fit together. (I'm similarly inept at keeping track of a bunch of different mechanical elements as a player.) I'd seen some previous discussions this time last year about using flow charts as a play aid, so I gave them a shot as an actual design aid.

I wrote up all the individual bits of the system for Do: Pilgrims of the Flying Temple on index cards and arranged them into some semblance of order.



And began translating those loose notes into a flow chart.



Several iterations followed, each one amputating or grafting organs as necessary. The actual system became its own page. Character creation became another. World creation became another, too.



Fast-forward to now, I've got a relatively complete multiple-page PDF of the whole system available for download here.

I know this method of game design isn't practical, or even necessary, for most folks. But, if you're anything like me, this method could be very helpful. I feel more comfortable going to a text-based format now that I can see the whole system at a glance. Each panel of the flow chart is its own page or double-page spread. Each color-coded section of the flowchart is a chapter.

These flowcharts are meant to be a design aid to better help me keep track of the system, so I'm not bothered if they're a little hard to follow as a player for now. I thought some other folks here would like to see a visual method of game design. I kind of like seeing a whole game's organs and guts splayed out across a single page.

    \n If you're doing this to assist with design, you might want to take a look at the "Swim Lane" format used by system designers. Its pretty much tailored to outlining processes being handed off between multiple parties.
    \n
    \nYou basically have a series of horizontal bands. Each band represents a "user", with one band reserved for "the system". So if you had a design that had a GM, a bunch of players, and a rotating player designated as "The Caller" you'd have 4 horizonal bands: "Game Mechanics", "GM", "Any Player", "The Caller".
    \n
    \nYou'd read the chart from left to right with actions being based back and forth between the bands, so you might have "declares action for their character" as a box in the "Any Player" swim lane, "decides whether or not the action is a Conflict" in "The Caller" swim lane, "Determines what roll to make and all modifiers" in "The GM" swim lane, and finally "Generates Degree of success / failure" in "The Game Mechanics" Swim lane.
    \n
    \nIts an especially useful format for keeping track of the order in which things need to happen and who has the authority over each thing. Being that its designed for developing human / computer interfaces it tends to be pretty rigid about having the precise order of things and precise hand off between user and system locked down. But for games that are pretty locked down it can be quite a useful visual tool.

    \n I haven't had much use for flowcharting GLASS--it's supposed to be simple and, more importantly, FAST--but I did mentally flowchart the Use of Abilities section, which yielded my current procedure (done in ordered list format, though now y'all got me thinking a basic flowchart wouldn't hurt, to supplement the textual rules).

    What I have done extensively is to try to cost-balance elements using a spreadsheet to help highlight the parallel elements of the system (Abilities, mainly) and make sure none are over- or under-priced or missing... or if there's a "hole' in the "grid of elements," then I know why it's there and might even address its absence in a sidebar in the appropriate spot in the rule (i.e. when the other parallel Abilities have been made evident).
