    \n Frankly, the ebook formats existing so far are absolutely not RPG friendly: they are perfect for novels and fiction in general (single column/flow of text, minimal text layout, rare illustrations) but very much NOT for something that is much more similar to a cross between an illustrated book and a technical manual like an RPG.

    Don't Rest Your Head by Evil Hat has been released on Kindle and other ebook formats recently and I'm sure it handles it pretty well because it *is* a single column text with minimal layout. Plus it's been converted by an expert.

    Then again, modern tablets can handle PDFs extremely well, so that's a very real alternative: rpg books look VERY nice on my Asus TF101 tablet (10 inch screen, 1280x800).
    And with software like ezPdf the user experience is much better than any reader software for pdfs I've experienced on computers. I can annotate the pdf, for example and I can even easily read double column Letter format pages on my PHONE (SGS i9000, circa 4 inch screen) thanks to its smart zooming.

    The future isn't very clear: a "next-generation-epub" format could handle more complex layouts, color e-ink devices should start to surface in the next couple of years... but at the same time, computing power *will* increase. And at that point that e-ink 10" color device will probably render pdfs just fine
    We'll see, we'll see...

    \n Considering the general issue, my own take is that rpg publishing needs to change to accomodate the likely technical paths book publishing is going to take in the future. This is mostly in reply to Renato there: the epub format and the HTML base it relies on will continue improving, and it seems likely that it'll be the basis of the universal text presentation standards of the future. Should this be the case I find value in starting to adapt our understanding of what a book is to the HTML format already; it's not too soon to start learning. It might be that some traditional elements of what makes a roleplaying game book, such as ample illustrations, might not make the transfer entirely intact. This is something to consider in detail, not a reason to dismiss the technology and count on PDF because it can replicate a paper book better.

    Already today I would like to argue that while an epub file can't be quite as nice-looking as a top-level book, it can be functional and pretty in somewhat different ways: effortless multilinearity provided by hypertext largely makes sidebars and such history, for example. Illustrations can be used, you just need to be critical of where in the text flow you want your illustration to be, and what purpose it serves. Also, epub is a transitional technology right now: I think that it won't be many years before the distinction between fully styled HTML and epub files becomes insignificant; the most current e-readers are already fully conversant with web pages. It is realistic to expect that soon you can make a portable book file with exactly the same range of tools you can use to create nice-looking websites today.

    PDF is also a transitional technology in the sense that we're just using it in digital publishing because of a lack of currently functional alternatives. As a long-term solution it's technically a dead end - it's basically a book-lookalike on computer screen, and making it anything else would entail rebuilding the entire file format from scratch. Better to work with something that's developed specifically with multi-platform digital consumption in mind, rather than print publication file wrapper, like PDF is.

    (In the spirit of clear communication I should emphasize that I don't say that Renato is wrong or wrong-headed, I'm just laying out my differing view on the likely future directions of the field.)

    Regarding Rafael's actual topic I sadly don't have any particularly relevant experiences. I'm more interested in using the open Internet as publishing medium, the walled gardens of the various ebook publishers don't seem that attractive to me. This is part of why the epub standard specifically appeals to me as technology; it's simple and compatible with existing web programming skills, so it offers a more egalitarian promise for ebook publishing than a model where you use proprietary tools from private companies to create different files for different markets. One of you prove me wrong by selling 10 000 copies of your game in the Apple store, though, and I'll be right behind you

    \n I've messed a bunch with Amazon formats for work, because we've been releasing some of our publications for the Kindle.

    Right now, there's no way I'd release a game in that format, because it's so astounding ugly and impossible to parse visually. Being able to read a text is one thing, but layout is super important in terms of indicating the relative importance of different pieces of information and providing an experience of reading and learning that is pleasant. I don't know how novelists can stomach releasing their books that way, honestly, and whoever invented the current epub format deserves to be jumped in an alleyway by a bunch of graphic designers. It would have been so easy to make a similar format that was actually pleasant to look at, but they intentionally decided not to, I guess. I have no idea why.

    That said, once we get a level of layout functionality in ereaders that at least as good as HTML was in the late 1990s, it may be something worth considering. I did a lot of web design in that era and, while a lot of it wasn't pretty, it was definitely possible to do some interesting and worthwhile things. Plus, I imagine there will be some huge leaps forward that jump quickly past that. Still, I wouldn't count Adobe out yet. I'm sure their designers can see the writing on the wall. Plus, there are so many documents available as PDFs that somebody will eventually come up with a PDF --> epub conversion that will preserve all the formatting. Anything else is crazy.

    \n Apple's free [on the Mac App Store] iBooks Author app is pretty nice: it's a customized version of their Pages word-processor for building eBooks for the iPad. Unfortunately it doesn't output straight ePub but a ".ibooks" format proprietary to their bookstore. But the books you can produce with it are pretty slick \u2014 the layout changes with device orientation, with landscape being more fancy and portrait more textual; you can include tables and semi-interactive graphics; and it supports hyperlinks and a glossary.

    I tried my hand at adapting the open-source "Free FATE" rules, and you can download the book here. Note that you need an iPad to use it. Drag the downloaded file to iTunes and then sync your iPad.

    Yeah, it's too bad that this is a proprietary format. I think there were some technical reasons (these books do some stuff ePub can't handle) but it's also Apple wanting to hog the books to themselves. On the other hand there's no way they would have released a slick free layout app for creating these if the results worked on anyone's e-reader.

    In the long run I think HTML is the way to go (as I understand it, ePub is basically just a subset of HTML 5.) The e-ink readers won't be able to support it very well, but those are probably doomed in the next few years, as Charles Stross convincingly argues. The more general-purpose tablets all tend to run WebKit-based browsers which have excellent CSS support.
