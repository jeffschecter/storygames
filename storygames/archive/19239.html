\n My friend and I are developing a story games database for a class project.

We are using RPG Geek as a model. We want to make something that is more specific to story games, and more searchable, but without all of the review/rating/forum features that RPG Geek has. We already have some ideas about the kinds of attributes we would like to add beyond what is available in RPG Geek and what we would like to not do, but I would like to ask a general question to everyone:

What problems do you have using RPG Geek? How could it be organized better for story games? And what kind of information about games would an ideal story games database include?

    \n Hey there! This is a project I've thought about a lot, in the "more than just a passing fancy, so I might as well register a domain name" type of way, but not an actual work way. I fully support you making something awesome. Maybe I'll work on something too down the line, maybe we'll work together. Who knows?

    Just from my thoughts and conversations I've already had on this:

    1) The most important thing, by far, I'd want this to do is answer questions like:

    "I've got five people and we want to play a freeform."
    "I've got five players plus me and we want to play something with no prep."

    I often just find myself sifting through my games folder over and over trying to remember which games fit which setups.

    2) One major hurdle is to think about what your inclusion criteria is: being encyclopedic is a lot of effort, and makes search more difficult (see #3) But once you start to limit down what's part of it, you get to all the fun (by which I mean, drama) of what is a story game? Do you accept playtests? Do you list every Game Chef game?

    In the end, I don't think there's an easy answer here.

    3) Search relevance rankings is actually really important. If I do the second search, and a nice picture of Lady Blackbird isn't in the first few images, then that's a bit weird. If GMless four player games don't include Fiasco near the top, also weird (not unjustifiable, just weird). But, coming up with search orders for searches is actually tricky. There's a lot of things you could do here, but it's an important thing to note that as the 'database' part gets more filled, the 'search' part gets harder to do well.

    4) I'd want browsing to be seamless and fun: other than that RPGGeek is hideous to look at it, it's also very poor at just promoting discoverability. If I was putting on my web design hat, I was thinking about search/profiles as a SPA (Single Page Application).

    Anyway, feel free to ignore all my advice. But that's my current thinking.

    \n I would like every user to be able to -tag- any entity in the database with any sort of tag they like. I'd like popular tags to be publicly seen (and voted on?). I think tagging can alleviate some of the search difficulties that James if foreseeing.

    I'd like users to be able to establish dynamic relationships between any database entities. Perhaps like tags, those relationships only become "true" when they reach a certain consensus-weight.

    And I agree that you should compile a list of use-cases in public and then design to satisfy all of them.

    \n Hi Everyone! Thanks for all of your great comments. We are including number of players as an attribute. I'm not sure how to include prep time, but we have 'time to play' and whether there is GM as attributes. Should 'prep time' be its own attribute?

If it goes well we would like to solicit more feedback and look into what would be required to continue with the project.
