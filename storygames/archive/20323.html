\n I just thought of this and maybe it's been obvious to everyone else all along, but...

There is this style of setting up a computer program called "MVC".
You have your "model" (the logic you are trying to calculate), the "view" (the interface) and the "controller" (which coordinates the two, speaks between the two).

Apocalypse World is set up around the conversation primarily. Its "model" is the fronts but a lot of logic is placed in the "view"; fiction [i]directly[/i] triggers moves. This is like putting a bunch of "if X then Y" stuff directly in your button code. (Please don't take this as a complaint on AW, which I think is an amazing design.)

There are a couple of other "models" beside the fronts, such as the player's current resources (hold, hx, barter etc) and of course, the big one: [i]what has been established[/i] in the shared imagined space.

But a lot the mechanics concern the conversation in and of itself.

Contrasted with D&D which has tons of models. The initative order, the action economy, the random encounter table, the turn structure (from older editions with the ten minute turn), the hex map, the prepared dungeons, the monster manual etc.


This isn't meant to explain Vincent's own game to him... it's pretty clear that he "gets" this stuff. Maybe it's more of a way for me to explain it to myself.

    \nWhen I think of the MVC model outside of programming contexts, I try to remain at a very high level, regardless of game. Typically (for a trad setup anyway) I see the Model as the GameWorld (part system, part GM, part session prep, part campaign notes or sourcebooks, part randomly determined and part improvised); the View is the Shared Imagined Space (which includes not only things the GM says but also things the genre assumes and portions of the gameworld for which Players are responsible), plus everybody's actions; the User is the Player(s) and the Controller is mostly the GM but also partly the Players (some functions - especially chargen - don't require GM intervention and instead directly access the Model, while some Player decisions are strictly color and directly affect the View without requiring a pass through the Controller-Model loop).

    I find that you can draw lines for various individuals in dividing up that structure, but in drawing the diagram for a particular game you answer all the questions you had, and then the diagram is no more useful than the praxis itself.

    \n In an RPG the controller is the dice. The view is the sheets and maps and the model is the rule book. At least that is how I would think of it. Controlling the game is through dice, conversations. The view is what the players experience, or the maps, figs, descriptions the GM reads. The model is the rules and data so those are the rule books, modules etc. character sheets I guess might be considered a model of a character since it is all the data about the character, but it is also a view since you can download a hundred different character sheets all for the same stats.

    Rule books are also partially controllers since they contain the rules of how the dice are used and how actions are resolved.

    But What is the equivalent for a RPG?

    I would say: Theme, definitions(objects), rules and sheets(descriptions). That is a good starting point.

    Theme is obviously the type of game and the feel that needs to flow through entire thing. Similar or description.

    Definitions are how things are defined. So stats, AC, saving throws, points etc. how a character, monster, ship, planet or door is defined.

    Rules are how objects(definitions) in the game interact...hmmm... Maybe objects instead of definitions. These are the actual turns, dice and difficulty charts that players work with. GM is also a player in this context.

    Sheets are the views. yes a bad name, but deals with figs, maps, sheets, handouts and descriptions that are used to present the rest of it.

    \n Dice don't actually make decisions about anything. Dice just produce random numbers.

    The controller in MVC is what decides what goes into the view, using the model as input, and storing state back into the model. Users interface only with the view.

    So in an RPG, who are the users? The players (and GM).

    What do the players actually interface with? The fiction (shared imagined space) of the game. I'd argue that they interface with dice and character sheets, too, but since the players are also the computing engine in this analogy, they act as the engine for the model, and dice and character sheets are model because they either produce calculations (a random number) or record state (the state of a character).

    Other things in the model: the GM's notes or module, the game text, the configuration of people around the table, note cards on the table, beads or other markers on various things, and discussions happening around the table.

    So players have discussions about what goes into the fiction by referencing their character sheets, the rules, various dice rolls, and other physical parts of the game. These discussions--via some agreed procedures (IIEE, if you care), and a kind of creative filter (each player's various agenda)--turn into accepted changes to the fiction (shared imagined space, or if you will allow me to change one of the terms, the "shared imagined state").

    Though the fiction is a "state," don't mistake this for model. The user interface for a web application has state, too. What is showing on the screen at any given time? Has the user clicked the OK button yet? The fiction is the view, because its state is the primary interface to the whole reason you play, which is to imagine a world together.
