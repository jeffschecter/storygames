    \n Yeah. I was casting about for a randomizing agent that we could use in a browser without resorting to a die roller. We used to use google page rank searches during BW editing marathons to prove which usage of a term is more popular and thus correct. So I suppose I owe Alexander for the inspiration.
    \n
    \nObviously, it's problematic. The values for the player's attributes become known after each is used. And the "GM" player can choose deliberately broad search terms to win a challenge. (Murder holes? Come on.)
    \n
    \nThe trick is to institute a rule that makes the search terms as florid and obscure as possible.
    \n
    \nThe range of results is too wild to use the margin of difference between the two, but I'm itching to try to use the actual page results in play. Somehow tie resolution into the first page that turns up.
    \n
    \nI'm also trying to figure a way to use "I'm feeling lucky." I'm thinking that maybe a quoted search term in that mode that turns up a result is a critical hit or special effect or something.

    \nIdea #1

    Do a "mod 9" calculation to get a number from 1-9 for each search and use those somehow. Mod-9 is easy to calculate in your head. Just add up the digits pairwise. Whenever your sum exceeds 1 digit, fold its digits by adding them, etc.

    Example: 349859456
    (3+4)+9+8+5+9+4+5+6 =
    (7+9)+8+5+9+4+5+6 =
    (16)+8+5+9+4+5+6 =
    (7+8)+5+9+4+5+6 =
    (15)+5+9+4+5+6 =
    (6+5)+9+4+5+6 =
    (11)+9+4+5+6 =
    (2+9)+4+5+6 =
    (11)+4+5+6 =
    (2+4)+5+6 =
    (6+5)+6 =
    (11)+6 =
    (2+6) =
    8

    Check:
    349859456 / 9 = 38873272.88888...
    349859456 - (38873272 * 9) =
    349859456 - 349859448 =
    8

    If you knew all that already, sorry.

    Idea #3

    In any case, that might have too much handling time for what you're using it for.

    Perhaps just use the last two (least significant) digits of the search results. If you get 349859456 results, that gets you a 56. Probably too hard to "game" and it becomes noise at that significance in a wide search.

    You might also ask the player to choose one word from the aspect you're searching, and each word can be used only once per session. Give players a 12-word budget or something to create their characters.

    \n I don't find the static nature of the player character aspects that big of a problem - definitely not something that would require other things to be compromised. Rather, you might think of those PC aspects as touchpoints that give meaning to the otherwise arbitrary numbers that spring up. The only way to tell if a million hits is a big or small number in this system is if you have some constant to compare with - and if you know that a million hits is enough to beat aspects A and B of the hero but not C, then that seems like a good thing to me. Of course there needs to be some reasoning that limits the player from always picking his best aspect in every conflict, but that's an old problem with a zillion solutions.

    (Probably still a good idea to make the aspects only semi-permanent, just to mix things up a bit. But changing them after each conflict seems unnecessarily random to me.)

    Anyway, an excellent way of making use of the microdungeons. You can feel the imaginative pressure in the play account. Been a while since I've been excited by one of these little design ideas in the Internet.

    \n So, you could ask the player to add an extra descriptor tag to her item/advantage every time she uses it. If she fails, she can use that extra descriptor again. If she succeeds, she cannot. Or, you always add the action (the name of the verb) to the item/advantage. You'll know what the baseline for a search term is, but adding something different to it each time puts in some variability.

    \n Random idea for using "I feel lucky":

    1) Enter your search term and click "I feel lucky".
    2) Look at the title bar of the result. Find the first full word in the title that was not one of the search terms.* This word is the result.
    3) Compare resulting words, highest alphabetically sorted word wins.
    4) The next time you use that same search term, you must subtract the result word you got previously.

    Example:

    - Lucky search "arcane amulet" (without quotes).
    - At the moment, this returns a 404 page! So, with the title bar saying "404 - Not Found", the first full word is "not".
    - Next time, lucky search "arcane amulet -not".
    - Title is "Arcane Amulet of the Sun...", so word is "of".
    - Next time, lucky search "arcane amulet -of".
    - and so on.


    * If there is a found page that contains no qualifying result word, the player narrates something happening to the aspect that results in its search phrase changing. Ideally, the contents of the page that came up would color the result, somehow. As an example, maybe it becomes "blackened arcane amulet" because first thing the player notices about the page is that it is very dark, and narrates how, during the challenge, flesh seared against the amulet, leaving an oily residue. The new phrase is then lucky searched and that result is used for the current challenge. If, on the other hand, the search initially returns no results, then the same thing happens, but the GM chooses the narration and term alteration based on the result of his own test.
